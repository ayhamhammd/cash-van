# دليل البدء والتشغيل واختبار كل الحالات — FlowVan + ERP

> دليل عملي لمحاكاة **تركيب جديد من الصفر** وتجربة جميع الحالات: إنشاء الحسابات، ربط
> FlowVan بنظام ERP، إعداد المناديب، ثم اختبار البيع والتحصيل والتسوية والذمم مع النتائج
> المتوقعة لكل خطوة.

الأنظمة الأربعة:
| النظام | الوصف | المنفذ |
|---|---|---|
| **الواجهة الخلفية (Backend)** | `cash-van-dashboard` (NestJS) — دماغ FlowVan | 3100 |
| **لوحة التحكم (Dashboard)** | `cash-van-dashboard-frontend` (Next.js) | 3000 |
| **نظام ERP** | `erp-saas` (Next.js) — الأصناف + المحاسبة | 3001 |
| **تطبيق الموبايل** | `FlowVan` (Kotlin Multiplatform) — المندوب | — |

## 1) (اختياري) تصفير البيانات لمحاكاة تركيب جديد

يبقي **الأصناف والعملاء والمؤسسة**، ويحذف الحركات والفواتير والقيود والمخازن ومخزون المناديب
حتى تبدأ من صفحة بيضاء. **نفّذه فقط إذا أردت بداية نظيفة** (سيمحو بيانات الاختبار الحالية).

```sql
-- على قاعدة بيانات ERP (erp_flowvan). خُذ نسخة احتياطية أولاً.
BEGIN;
-- حركات محاسبية
DELETE FROM journal_entry_lines;
DELETE FROM journal_entries;
-- مبيعات/مقبوضات
DELETE FROM invoice_payments;
DELETE FROM invoices;
DELETE FROM financial_transactions;
-- مخزون + مخازن المناديب (الڤانات)
DELETE FROM stock_movements;
DELETE FROM item_stock;         -- أرصدة المخزون لكل مخزن
DELETE FROM warehouses WHERE is_van = true;   -- مخازن المناديب فقط
-- سجل الربط مع FlowVan (يمنع اعتبار الدفعات مكرّرة)
DELETE FROM external_document_refs WHERE source_system = 'van_sales';
COMMIT;
```
وعلى قاعدة بيانات FlowVan (`flowvan`) لتصفير صندوق الإرسال والتسويات إن رغبت:
```sql
DELETE FROM erp_outbox;
DELETE FROM salesman_settlement;
```
> **الأصناف تبقى**: لم نحذف جدول الأصناف (`product_skus` / `item_cart`) — «فقط الأصناف على ERP».

## 2) إعداد نظام ERP (الحسابات وقواعد الترحيل)

### 2.1 دليل الحسابات
من **ERP ← المحاسبة ← دليل الحسابات** (أو معالج القوالب `Chart Templates`). أنشئ الحسابات
التالية (الرموز اقتراحية؛ المهم **المفتاح النظامي** `system_account_key`):

| الرمز | الاسم | النوع | المفتاح النظامي |
|---|---|---|---|
| 1101 | ذمم العملاء | أصول Asset | `ACCOUNTS_RECEIVABLE` |
| 1002 | النقدية والبنك | أصول Asset | `BANK` |
| 2002 | ضريبة المبيعات المستحقة (مخرجات) | التزامات Liability | `TAX_PAYABLE` |
| 4001 | إيرادات المبيعات | إيرادات Revenue | `SALES_REVENUE` |

وحسابات التسوية والمناديب (حسب رغبتك — يمكن استخدام معالج القوالب):
| الرمز | الاسم | النوع | الدور |
|---|---|---|---|
| 301  | حساب المندوب (عهدة) — لكل مندوب حساب | — | يحمل عجز المندوب |
| 302  | حساب المبيعات (تسوية/مقاصة) | — | الطرف الدائن في قيد التسوية |
| 503  | حساب التحصيلات النقدية (شركة) | أصول | وجهة النقد المُستلَم |
| 504  | حساب تحصيل الشيكات (شركة) | أصول | وجهة الشيكات |

### 2.2 قواعد الترحيل (Posting Rules)
من **ERP ← الإعدادات ← قواعد الترحيل** تأكد أن:
- **SALES_INVOICE** = تلقائي عند الحفظ (`AUTO_ON_SAVE`)، وقالبه: مدين `ذمم العملاء` بالإجمالي /
  دائن `إيرادات المبيعات` بالصافي / دائن `ضريبة المبيعات` بالضريبة.
- **RECEIPT_VOUCHER** = تلقائي (`AUTO_ON_SAVE`)، وقالبه: مدين `البنك` / دائن `ذمم العملاء`.
> إن لم تُضبط، لن تنكسر العملية: الفاتورة/السند يُنشآن، ويُتخطّى القيد مع تحذير (تحصين مطبّق).

### 2.3 مفتاح API والصلاحيات
من **ERP ← الإعدادات ← مفاتيح API** أنشئ مفتاحًا لـ FlowVan بالصلاحيات:
`products:read` · `customers:read/write` · `sales_invoices:read/write` · `payments:read/write` ·
`accounting:read` · **`accounting:write`** · `warehouses:read` · `organization:read` ·
`price_lists:read`. **انسخ المفتاح** (يظهر مرة واحدة).
> غياب `accounting:write` = القيود تُرفض بـ `FORBIDDEN`. غياب صلاحية = 403.

## 3) ربط FlowVan بنظام ERP

من **لوحة التحكم ← الإعدادات ← ERP**:
1. فعّل «العمل مع ERP».
2. ضع عنوان ERP: `http://localhost:3001` والمفتاح المنسوخ.
3. اضغط **اختبار الاتصال** — يجب أن ينجح.
4. فعّل «التصدير المباشر» (Direct Export) ليُرسل البيع/التحصيل تلقائيًا.

انتظر دورة المزامنة (≈60 ثانية) أو اضغط «مزامنة الآن» ليسحب FlowVan **الأصناف** من ERP.

---

## 4) إعداد الحسابات داخل FlowVan

1. **الإعدادات ← المحاسبة**: عيّن الحسابات الثلاثة:
   - حساب المبيعات (تسوية) = `302`
   - حساب التحصيلات النقدية = `503`
   - حساب تحصيل الشيكات = `504`
2. **المناديب ← تعديل مندوب ← حساب المندوب (ERP)** = حساب عهدة المندوب (مثلًا `301`).

## 5) مصفوفة اختبار كل الحالات

> بعد كل حالة، تحقق من: (أ) لوحة FlowVan، (ب) صندوق الإرسال `outbox` بحالة `posted`،
> (ج) اليومية في ERP. أمر فحص صندوق الإرسال:
> ```bash
> curl -s 'http://localhost:3100/api/v1/erp/outbox' -H "Authorization: Bearer $TOKEN" \
>   | jq '.data[] | {kind,status,error,ref}'
> ```

| # | الحالة | الخطوة | النتيجة المتوقعة |
|---|---|---|---|
| 1 | **بيع نقدي** | من الموبايل: بيع نقدي لعميل | فاتورة في ERP؛ قيد `SALES_INVOICE`: مدين ذمم / دائن مبيعات+ضريبة؛ رصيد العميل يرتفع |
| 2 | **بيع آجل** | بيع بالآجل (Credit) | كسابقه؛ **ذمم العميل ترتفع** بقيمة الفاتورة |
| 3 | **تحصيل نقدي** | تحصيل مبلغ من العميل | يُخصم من أقدم الفواتير (FIFO)؛ قيد `RECEIPT_VOUCHER` مدين بنك / دائن ذمم؛ **الذمم تنخفض** |
| 4 | **تحصيل بشيك** | تحصيل بشيك | مثل النقدي — **الذمم تنخفض فورًا** (الشيك يُعامَل كالنقد حسب الإعداد) |
| 5 | **تحصيل زائد** | تحصيل أكبر من مديونية العميل | يُسدَّد كل المستحق؛ الفائض يُحفظ كـ **سلفة/رصيد دائن** يُخصم من الفاتورة القادمة |
| 6 | **تسوية نهاية اليوم** | لوحة ← نهاية اليوم ← تسوية المندوب (أدخل المبلغ المُستلَم) | قيد «تسوية مندوب المبيعات (الاسم) — التاريخ»: مدين تحصيلات نقدية + شيكات + (عجز على حساب المندوب) / دائن حساب التسوية — **متوازن** |
| 7 | **عجز في التسوية** | المُستلَم أقل من المتوقع | الفرق يبقى **مدينًا على حساب المندوب** (يُرحَّل لليوم التالي) |
| 8 | **التحقق من الذمم** | لوحة ← الذمم (AR) | الرصيد يعكس: مبيعات آجلة − تحصيلات؛ يتطابق مع رصيد ERP |

### أوامر تحقق سريعة (على قاعدة ERP)
```sql
-- رصيد عميل = Σ(إجمالي الفواتير − المدفوع)
SELECT c.name, SUM(i.total_amount - i.amount_paid)/1000.0 AS balance
FROM invoices i JOIN customers c ON c.id=i.customer_id
WHERE i.status <> 'voided' GROUP BY c.name;

-- توازن قيد التسوية
SELECT je.description, SUM(l.debit)/1000.0 dr, SUM(l.credit)/1000.0 cr
FROM journal_entries je JOIN journal_entry_lines l ON l.journal_entry_id=je.id
WHERE je.description LIKE '%تسوية مندوب%' GROUP BY je.id, je.description;
```

## 6) استكشاف الأخطاء

| العطل | السبب | الحل |
|---|---|---|
| صف `outbox` بحالة `dead_letter` — `FORBIDDEN` | المفتاح ينقصه `accounting:write` | أضف الصلاحية في ERP ثم أعد المحاولة |
| `dead_letter` — `HTTP 500` عند البيع | لا يوجد قالب ترحيل + الوضع تلقائي | اضبط قالب `SALES_INVOICE` أو اجعله يدوي؛ (التحصين يمنع الانهيار بعد التحديث) |
| صف `CASH_SETTLEMENT` قديم عالق | نوع قديم مهجور | تجاهله؛ لم يعد يُنتَج |
| رصيد العميل لا ينخفض بعد التحصيل | تحصيل «على الحساب» لم يُخصم | مطبّق في Layer 1: يُخصم FIFO ويُرحَّل قيد القبض |
| «هذه البيانات تُدار بواسطة ERP» | بيانات رئيسية للقراءة فقط في وضع ERP | أنشئ/عدّل في ERP؛ (حسابات التسوية تُضبط من مسار غير محمي) |
| إعادة المحاولة | زر إعادة المحاولة أو: | `curl -X POST '…/erp/outbox/<id>/retry' -H "Authorization: Bearer $TOKEN"` |

## 7) ملاحظة عن مخزون الڤان (Van stock)

FlowVan هو **مصدر مخزون الڤان**. إن أردت ألا يتتبّع ERP كميات مخازن المناديب (كما في تصفير
الخطوة 1)، فالبيع سيظل يُنشئ **الفاتورة والقيد المحاسبي** في ERP لكن دون تحريك مخزون مخزن.
هذا قرار تشغيلي — أخبرني إن رغبت بتعطيل إنشاء مخازن المناديب ودفعات المخزون في الكود.
